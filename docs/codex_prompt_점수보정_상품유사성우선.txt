이번 작업의 핵심 목표:
현재 프로그램의 가장 큰 문제는, 사용자가 선택한 지정상품/유사군코드와 실질적으로 관련 없는 선행상표가 있어도 점수에 과도하게 반영되어 최종 등록가능성이 지나치게 낮아지는 점이다.
이를 고쳐서, 실제 상표 심사 실무에 더 가깝게 다음 순서로 판단하게 만들어라.

최우선 판단 순서:
1. 절대적 거절사유(식별력) 판단
2. 상품 유사성 필터링
   - 동일 유사군코드
   - 동일 류
   - 경제적 견련성 있는 타 류
   - 무관한 타 류는 제외
3. 상품 유사성 필터를 통과한 후보에 대해서만 표장 유사성 판단
   - 외관
   - 호칭/발음/명칭
   - 관념
4. 혼동 가능성 종합
5. 최종 점수 캘리브레이션

가장 중요한 수정 원칙:
- 기존 로직은 최대한 유지한다.
- 특히 아래는 유지한다.
  - 현재 similarity_percent 및 문자열 유사도 관련 흐름
  - 발음/명칭/호칭 유사도 판단 로직
  - search_mapper.py의 상품/서비스 추천 로직
  - similarity_code_db.py의 유사군코드 추천 로직
  - KIPRIS 검색 흐름
  - Streamlit 단계형 UI 흐름
- 이번 작업은 “전면 재작성”이 아니라 “판단 순서 수정 + 점수 합성 방식 보정 + 설명 고도화”가 목적이다.

반드시 고쳐야 할 핵심 버그:
- 사용자가 3류를 선택했는데 25류 상표만 검색돼도 최종 점수가 과도하게 낮아지는 문제
- 사용자가 20류 가구를 선택했고 상품 유사성 필터를 통과한 선행상표가 0건인데도 점수가 60점대 후반으로 머무는 문제
- 즉, “검색 결과가 있음”과 “실제 충돌 후보가 있음”이 구분되지 않는 문제

수정해야 할 판단 구조:
A. 선행상표 후보를 4개 그룹으로 분리하라.
1) group_exact_code
   - 동일 유사군코드
   - 가장 강한 충돌 후보군
2) group_same_class
   - 동일 류이나 유사군코드는 다름
   - 보조 검토군
3) group_related_market
   - 타 류이나 경제적 견련성 있음
   - 예외 검토군
4) group_irrelevant
   - 타 류, 타 코드, 경제적 견련성 없음
   - 원칙적으로 최종 점수 계산에서 제외

B. group_irrelevant는 이렇게 처리하라.
- 전체 검색 건수에는 포함 가능
- 그러나 top_prior, 상대적 거절사유, 최종 점수 감점에는 반영하지 마라
- UI와 보고서에서는 “검색은 되었으나 상품 유사성 검토에서 제외된 후보”로만 설명하라

C. 상품 유사성 필터가 최우선 게이트가 되게 하라.
- 현재처럼 류가 안 겹쳐도 일괄적으로 강한 감점이 들어가는 구조는 없애라
- 타 류/타 코드 후보는 원칙적으로 0 감점
- 다만 아래 예외는 허용
  - 동일 유사군코드인 타 류
  - 경제적 견련성이 큰 경우
  - 주지·저명상표
  - 기만/부정한 목적 가능성
- 이 예외도 기본 로직이 아니라 보조 경고 수준으로 처리하라

D. 최종 점수 합성 방식을 보정하라.
현재는 “선행상표 없음”이 약한 가산점처럼 작동하는 문제가 있다.
이를 아래 방향으로 바꿔라.

1) 상품 유사성 필터 통과 선행상표가 0건이면
- 상대적 거절사유 리스크를 매우 낮게 본다
- 최종 점수는 식별력 중심으로 높은 구간에서 형성되게 한다

2) 캘리브레이션 기준을 아래처럼 적용하라.
- 식별력 보통 이상 + 필터 통과 선행상표 0건:
  최종 점수 기본 범위 82~90
- 조어상표 + 필터 통과 선행상표 0건:
  최종 점수 기본 범위 88~95
- 식별력 약함 + 필터 통과 선행상표 0건:
  최종 점수 기본 범위 72~82
- 동일 유사군코드 충돌 후보 존재 + 표장 유사도 높음:
  50 이하까지도 가능
- 동일 류이나 비유사 코드 + 표장 유사도 중간:
  55~75 범위의 중간 리스크
- 타 류/타 코드 후보만 존재하고 필터 통과 후보 0건:
  그 자체만으로 70 미만으로 떨어지지 않게 하라
  단, 식별력이 심각하게 약한 경우는 예외

3) 판매업 코드 관련 점수 보정
- 단순히 S 코드가 선택되었다는 이유만으로 감점하지 마라
- 동일 판매업 코드에서 실제 충돌 후보가 있을 때만 제한적으로 감점하라
- 판매업 코드 선택은 보호 범위 확대 목적도 있으므로 기본적으로 중립 또는 약한 주의 신호로 처리하라

E. 식별력 판단은 기존보다 더 세분화하되, 선행상표 리스크와 분리하라.
아래 항목은 유지/보강하라.
- 보통명칭
- 관용표장
- 성질표시
- 현저한 지리적 명칭
- 흔한 성 또는 명칭
- 간단하고 흔한 표장
- 기타 식별력 없음
하지만 식별력 약함만으로 자동으로 60점대 이하가 되지 않도록 하라.
즉, 식별력 리스크와 선행상표 충돌 리스크를 별도의 축으로 유지하라.

F. UI/결과 표시를 수정하라.
step 4 결과 화면에서 아래 수치를 분리해서 보여라.
- 전체 검색 건수
- 상품 유사성 필터 통과 건수
- 실제 충돌 위험 건수
- 제외된 후보 건수
또한 아래 설명을 추가하라.
- 왜 일부 검색 결과가 최종 점수에 반영되지 않았는지
- 왜 필터 통과 선행상표가 없으면 점수가 높게 나오는지
- 왜 식별력 약함이 있더라도 충돌이 없으면 등록 가능성이 여전히 높을 수 있는지

G. PDF 보고서도 동일한 구조로 바꿔라.
반드시 아래 섹션을 넣어라.
- 식별력 판단
- 상품 유사성 검토 결과
- 표장 유사성 검토 결과
- 혼동 가능성 종합
- 점수 산정 해설
특히 “검색 결과가 있었지만 상품 유사성 검토에서 제외되었다”는 설명을 보고서에도 넣어라.

H. 테스트 케이스를 추가하라.
반드시 아래 테스트를 만든다.

1) Lexai / 20류 / 가구 / G2001 + S2021
- 전체 검색 결과는 있을 수 있음
- 그러나 상품 유사성 필터 통과 선행상표 0건이면
- 최종 점수는 82 이상이 되게 하라
- 식별력이 특별히 나쁘지 않다면 85 전후도 허용

2) Lexai / 9류 / 소프트웨어
- 동일 또는 유사한 선행상표가 존재하면
- 20류보다 유의미하게 낮게 나와야 한다

3) 3류 화장품 선택 / 25류 의류 선행상표만 존재
- 경제적 견련성이 없으면
- 최종 감점에 거의 반영하지 마라

4) 동일 유사군코드 + 높은 호칭 유사도
- 강한 충돌로 판단되게 하라

5) 타 류이지만 동일 유사군코드인 예외 케이스
- 검토 대상에 포함되게 하라

J. 구현 세부 원칙
- 기존 함수를 가능하면 유지하고, 보정 레이어를 덧붙이는 방식으로 구현하라
